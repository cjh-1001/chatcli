# x64dbg Runtime Analysis Skill

Plan and review x64dbg-assisted runtime analysis without turning a suspicious sample into unsafe host execution. Use this skill to make ChatCLI generate x64dbg scripts from its own static artifacts, choose a breakpoint coverage mode, and map debugger observations back to static evidence.

## Scope And Safety

Use x64dbg only when all are true:

1. The target is authorized.
2. The run is inside an isolated local lab, VM, sandbox, CTF/crackme environment, or owned test target.
3. Dynamic execution is explicitly in scope.
4. Rollback or disposal is available for unknown or suspicious samples.

Do not run unknown samples on the analyst host. If isolation, authorization, or rollback is unclear, stop before execution and ask for the missing boundary. For broad malware behavior triage, start with `malware-triage`; for remote Tencent Cloud dynamic collection, use `tencent-remote-analysis`.

Allowed outcomes:

- debugger breakpoint plan
- runtime string or config extraction plan
- x64dbg script generated by `runtime_string_hooks`
- interpretation of existing x64dbg logs, dumps, screenshots, or analyst notes
- comparison between static hypothesis and observed runtime values

Disallowed outcomes:

- improving malware stealth, persistence, credential theft, evasion, or destructive behavior
- guidance for operating against third-party systems
- bypass instructions whose purpose is to help malware evade real defenses
- claiming behavior is confirmed without a real debugger log, dump, trace, or other runtime artifact

## Related Skills

- Read `../malware-triage/SKILL.md` first when the user asks for broad malware behavior, IOC extraction, or a report from a suspicious sample.
- Read `../reverse-audit/SKILL.md` first when the user asks for function-level reverse analysis, CTF/crackme validation logic, patch audit, or existing conclusion review.
- Read `../common/references/tool-registry.md` when tool names are unclear. The registered ChatCLI tool is `runtime_string_hooks`; `x64dbg` itself is an external debugger reported by tool health or the Guest Agent inventory, not a ChatCLI tool name.
- Read `../tencent-remote-analysis/SKILL.md` when the sample is on the Tencent Cloud analysis server. x64dbg is not part of that baseline dynamic collector stack.

## Workflow

### 1. Classify The x64dbg Task

Choose one route:

| Route | Use when | Next action |
| --- | --- | --- |
| Runtime strings/config | Decrypt routine, API return value, encoded strings, runtime-only config | Use `runtime_string_hooks` to generate Frida and x64dbg scripts |
| Breakpoint plan | User asks where to break or what to inspect | Derive breakpoints from static offsets, imports, xrefs, or decompiled calls |
| Existing debugger artifact review | User provides logs, dump paths, screenshots, register snapshots, or notes | Interpret artifacts and map them to claims/confidence |
| Anti-debug observation | Static evidence includes debugger checks or runtime divergence | Document checks and observation goals; avoid real-world evasion enhancement |
| Broad dynamic behavior | User wants process/file/registry/network telemetry | Hand off to `malware-triage` or `tencent-remote-analysis` |

### 2. Autonomous Generation Decision

Follow this order. Do not skip to manual API lists unless the prior step is impossible.

1. If a ChatCLI static result directory exists, call:

   ```text
   runtime_string_hooks analysis_dir=<result-dir> output_dir=.chatcli/tmp/hooks/<case-id> coverage_mode=<mode>
   ```

2. If the user wants the script on a remote analysis VM, add:

   ```text
   sync_to_remote=true remote_output_dir=<server-dir>
   ```

   This only copies scripts and plans; it must not execute the sample or start x64dbg.

3. If no static directory exists, first obtain static artifacts through the appropriate skill/tool chain. For Tencent Cloud samples, use `tencent-remote-analysis` to produce/download static results. For local artifacts, use `malware-triage` or `reverse-audit` static tools.

4. If a specific decrypt/config function is already proven, pass `module_name` and `decrypt_offset`. Otherwise leave `decrypt_offset` empty and let the tool infer API breakpoints.

5. Always inspect `chatcli_x64dbg_plan.json` before summarizing coverage. Treat the plan as the source of truth for ranked breakpoints, coverage mode, evidence files, Go clues, and coverage gaps.

### 3. Choose Coverage Mode

| Mode | Use when | Required output |
| --- | --- | --- |
| `focused` | The user wants a short manual x64dbg session or one behavior family | top breakpoints and why lower-priority hooks were omitted |
| `balanced` | Default analyst workflow | category summary plus highest-ranked breakpoints |
| `comprehensive` | The user asks whether attack behaviors are broadly covered | full category coverage and explicit noise/gap warning |

Default to `balanced`. Use `comprehensive` only when coverage is more important than debugger noise, or when the user asks whether behaviors are fully covered.

## Behavior Coverage Matrix

Use the generated x64dbg plan as targeted breakpoint coverage, not as complete
runtime telemetry. Prefer these breakpoint categories:

| Behavior | x64dbg coverage | Evidence to capture | Complementary telemetry |
| --- | --- | --- | --- |
| Scheduled task persistence | `CreateProcess*`, `CoCreateInstance` | `schtasks` command line or Task Scheduler COM object | Procmon process/registry |
| Service persistence | `OpenSCManagerW`, `CreateServiceW`, `StartServiceW`, `ChangeServiceConfigW` | service name, binary path, start type | Procmon/Sysmon service events |
| Registry persistence/config | `RegCreateKeyExW`, `RegSetValueExW`, `RegQueryValueExW`, delete APIs | key path, value name, data buffer | Procmon registry events |
| File staging/impact | `CreateFileW`, `ReadFile`, `WriteFile`, `DeleteFileW`, `MoveFile*`, `FindFirstFileW` | paths, access mode, written buffers | Procmon file events |
| Process injection | `OpenProcess`, `VirtualAllocEx`, `WriteProcessMemory`, `CreateRemoteThread`, `NtCreateThreadEx`, `QueueUserAPC` | target PID, allocation size, written buffer, thread start | Sysmon/process telemetry |
| Dynamic import/API hashing | `LoadLibrary*`, `GetProcAddress` | DLL name and API name | static xrefs and decompile |
| Network/C2 | `WSAStartup`, `WSASocketW`, `connect`, `send/recv`, WinHTTP/WinINet APIs, `DnsQuery_W` | endpoint, request path, sent/received buffers | PCAP/TShark/Zeek |
| Crypto/decryption | CryptoAPI/BCrypt, `CryptStringToBinaryW`, `CryptUnprotectData` | buffer pointers, sizes, return buffers | memory dumps/static data flow |
| Credential access | `CryptUnprotectData`, `OpenProcess`, `ReadFile` | DPAPI inputs, target process/path, output buffer | Procmon/Sysmon, memory evidence |
| Discovery | `GetUserNameW`, `GetComputerName*`, `NetUser*`, `NetShareEnum`, adapter APIs | returned host/user/network info | command/process logs |
| Collection | clipboard and screenshot APIs (`OpenClipboard`, `BitBlt`, `GetDIBits`) | clipboard handles, bitmap dimensions/buffers | screen/file artifacts |
| Anti-analysis | `IsDebuggerPresent`, `CheckRemoteDebuggerPresent`, timing/window/process checks | branch outcome and checked artifact | static classification; do not provide evasion enhancement |

Coverage gaps:

- x64dbg breakpoints do not prove behavior happened unless the breakpoint hits.
- API breakpoints can miss direct syscalls, inline assembly, custom protocol
  parsing, encrypted config, or behavior hidden behind trigger conditions.
- GUI x64dbg cannot replace Procmon/Sysmon/PCAP for process, file, registry, or
  network ground truth.
- For Go samples, use Go build/module/symbol clues to guide breakpoints, but do
  not treat Go runtime symbols as malicious by themselves.

### 4. Start From Static Evidence

Before a debugger plan, collect or reuse:

- module name and architecture
- entry point or function offset
- decrypt/unpack/config function candidates
- relevant imports such as `LoadLibrary`, `GetProcAddress`, crypto APIs, string APIs, or debugger checks
- expected argument registers or stack arguments
- expected output buffers, return values, files, registry keys, or network indicators

Prefer focused static tools before runtime instrumentation: `binary_inspect`, `encoded_string_extract`, `obfuscated_data_map`, `ida_analyze`, `ida_focus_decompile`, `ghidra_analyze`, and `reverse_technique_map`.

### 5. Generate Scripts When Needed

Use `runtime_string_hooks` when the task is to prepare runtime extraction scripts.
Prefer autonomous static-artifact audit when ChatCLI has already downloaded a
case result directory:

```text
runtime_string_hooks output_dir=.chatcli/tmp/hooks/<case-id> analysis_dir=.chatcli/remote_results/<case-id> coverage_mode=balanced
```

With `analysis_dir`, the tool must inspect `binary_inspect.json`, `exiftool.txt`,
`strings.txt`, and `floss.txt`, then infer:

- module name from the sample path
- entry point from ExifTool
- API breakpoints from imported/static strings and behavior keywords
- ranked breakpoint reasons and confidence from static evidence
- Go build/module/symbol clues when present
- x64dbg script lines from the built-in template
- `chatcli_x64dbg_plan.json` and `chatcli_x64dbg_usage.md`

Do not manually invent API hooks before trying this mode. Review the generated
metadata and only add extra `api_names` when the static evidence clearly
supports them.

When the debugger will be used on the Tencent Cloud server and `remote_guest` is
configured, set `sync_to_remote=true` and `remote_output_dir=<server-dir>` to
copy the generated scripts and plan to the remote analysis VM. This sync step
does not execute the sample or start x64dbg.

When a specific decrypt function or config routine is known, provide only known
values:

```text
runtime_string_hooks output_dir=.chatcli/tmp/hooks module_name=<target.exe> decrypt_offset=<0xoffset> api_names=<apis> arg_indexes=<indexes>
```

The generated x64dbg script is a starting point. It should be loaded manually in the authorized lab debugger. Do not describe script generation as execution.

If offsets are not known, produce a short plan for resolving them first instead of inventing offsets.

### 6. Interpret Debugger Evidence

For every x64dbg-derived claim, record:

- artifact: log, dump, screenshot, breakpoint hit, register snapshot, memory region, or analyst note
- where: module+offset, function name, API, or breakpoint condition
- evidence: register values, buffer contents, return value, decoded string, config field, or branch outcome
- interpretation: what the evidence supports
- confidence: `confirmed`, `high`, `medium`, `low`, `unsupported`, or `contradicted`
- gap: missing trigger, incomplete trace, no caller reachability, truncated dump, or environment divergence

Use `confirmed` only when the artifact directly shows the runtime value or branch outcome, or when static data flow proves reachability.

### 7. Required Final Summary

For script-generation tasks, always report:

- whether execution happened (`script generation only`, `x64dbg loaded`, or `sample ran`)
- `coverage_mode`
- generated local script path
- remote sync path and status, if used
- module name and entry offset
- breakpoint count and top categories
- strongest static clues, including Go module/build clues when present
- coverage gaps that require Procmon/Sysmon/PCAP or deeper reversing

### 8. Report Boundaries

State clearly whether:

- x64dbg execution was performed or only planned
- the environment was isolated
- static and runtime evidence agree
- any behavior remains unobserved because the breakpoint did not hit, the path was not triggered, or the artifact is missing

Do not use x64dbg output as a substitute for host/network telemetry. For process/file/registry/network behavior, require Procmon/Sysmon/PCAP/TShark artifacts or explicitly mark the behavior as not dynamically observed.
